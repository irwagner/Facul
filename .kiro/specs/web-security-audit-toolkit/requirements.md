# Requirements Document

## Introduction

The **Web Security Audit Toolkit** is a web application security auditing framework developed for academic use with written authorization from the institution. The tool aims to systematize the process of identifying, documenting, and reporting vulnerabilities in web applications, with special focus on modern stacks including Vue.js/Vite on the frontend, Nginx as the web server, Amazon CloudFront as CDN, and Web3 integration via Ethers.js.

The toolkit guides the auditor through a structured seven-phase flow — from attack surface discovery to technical report generation — prioritizing non-destructive, low-risk actions, with an iterative approach where the user brings results from each step for analysis and guidance.

## Glossary

- **Toolkit**: The web security auditing system described in this document.
- **Auditor**: The human user who executes the audit with written authorization.
- **Target**: The authorized web application being audited.
- **Finding**: A vulnerability or security issue identified during the audit.
- **Report**: Structured technical document generated at the end of the audit with all findings.
- **Scanner**: Component responsible for executing automated checks.
- **Analyzer**: Component responsible for interpreting results and classifying findings.
- **Reporter**: Component responsible for formatting and generating the final technical report.
- **Nuclei**: Open-source vulnerability scanning tool (projectdiscovery/nuclei) integrated into the Toolkit.
- **Burp_Suite**: HTTP proxy for manual validation integrated into the Toolkit flow.
- **Surface_Mapper**: Component responsible for the attack surface discovery phase.
- **Enumerator**: Component responsible for enumeration of subdomains, hosts, ports, and endpoints.
- **Fingerprinter**: Component responsible for identifying technologies and versions of active services.
- **Severity**: Risk level classification of a finding: low, medium, high, or critical.
- **Confidence**: Confidence level in the validity of a finding: low, medium, or high.
- **Source_Map**: File generated by Vite that maps minified code to original source code.
- **IDOR**: Insecure Direct Object Reference — unauthorized access to resources via identifier manipulation.
- **CSP**: Content Security Policy — HTTP header controlling resource origins.
- **HSTS**: HTTP Strict Transport Security — HTTP header that enforces HTTPS usage.
- **CDN_Bypass**: Technique for discovering the real server origin IP by circumventing the CDN.

## Requirements

### Requirement 1: Audit Authorization Control

**User Story:** As an academic auditor, I want to register the institution's written authorization before starting any test, so that I ensure all audit activity is legally and ethically supported.

#### Acceptance Criteria

1. WHEN the Auditor starts a new audit session, THE Toolkit SHALL request registration of the authorized target, including domain, institution name, and date of written authorization.
2. IF the Auditor attempts to execute any scanning module without having registered a valid authorization (defined as all three mandatory fields present and authorization date no more than 1 year before the current date), THEN THE Toolkit SHALL block execution and display a message indicating that authorization is required.
3. THE Toolkit SHALL store the authorization data in a session configuration file in the audit working directory; IF the write operation fails, THE Toolkit SHALL abort the session initialization and display an error message with the filesystem path and failure reason.
4. WHEN the Auditor executes any check, THE Toolkit SHALL validate that the target domain matches the authorized scope (exact domain match or subdomain suffix match against the pre-configured list of authorized domains and CIDR ranges) before dispatching any network request.
5. IF the Auditor provides a target outside the authorized scope, THEN THE Toolkit SHALL refuse execution and log the event in the audit log, including: timestamp, attempted target, authorized scope, and module that was requested.
6. WHEN an authorization expires (date exceeds 1 year from registration), THE Toolkit SHALL display a warning on session load and require the Auditor to confirm or update the authorization before proceeding.

### Requirement 2: Attack Surface Discovery

**User Story:** As an academic auditor, I want to map the complete attack surface of the target, so that I have visibility into all entry points before deepening the tests.

#### Acceptance Criteria

1. WHEN the Auditor starts the discovery phase, THE Surface_Mapper SHALL enumerate subdomains of the target using passive techniques (DNS queries, Certificate Transparency logs).
2. WHEN subdomain enumeration is complete, THE Surface_Mapper SHALL identify active hosts from the list of found subdomains, where an active host is defined as one that responds to a TCP SYN or ICMP echo within 5 seconds.
3. WHEN active hosts are identified, THE Enumerator SHALL perform port scanning on ports 80, 443, 8080, 8443, 8000, 8888, 9090, 9443, 3000, and 5000.
4. WHEN port scanning is complete, THE Fingerprinter SHALL identify technologies and versions of active services, including web server, framework, and CDN.
5. WHEN all prior discovery phases are complete, THE Surface_Mapper SHALL generate an attack surface map containing: list of subdomains found, list of active hosts with their IPs, list of open ports per host, and list of identified technologies per service.
6. IF a subdomain or host identified is outside the authorized scope (not matching any entry in the pre-configured list of authorized domains and CIDR ranges), THEN THE Surface_Mapper SHALL exclude it from the map and log the exclusion with: excluded host, reason, and timestamp.
7. IF subdomain enumeration yields zero results, THEN THE Surface_Mapper SHALL log a warning indicating that no subdomains were found and prompt the Auditor to verify the target domain and DNS resolution before proceeding.

### Requirement 3: Endpoint and Parameter Enumeration

**User Story:** As an academic auditor, I want to enumerate endpoints, directories, and application parameters, so that I identify the full API surface and routes before performing specific tests.

#### Acceptance Criteria

1. WHEN the Auditor starts the endpoint enumeration phase, THE Enumerator SHALL perform directory and file discovery using configurable wordlists with a minimum of 100 entries.
2. WHEN directory discovery is executed, THE Enumerator SHALL test paths of common administrative panels (e.g., /admin, /dashboard, /panel, /manage, /api/v1/admin) in addition to wordlist entries.
3. WHEN the application returns HTTP 200 responses for enumerated paths, THE Enumerator SHALL record the found endpoint with: status code, response body size in bytes, and page title extracted from the HTML `<title>` tag if present.
4. WHEN the application returns HTTP 301 or 302 redirects for enumerated paths, THE Enumerator SHALL record the redirect with: original path, status code, and `Location` header value.
5. WHEN the Enumerator finds API endpoints, THE Enumerator SHALL attempt to identify accepted parameters by analyzing response bodies for parameter names and varying one parameter at a time, recording the observable output field names from the response JSON.
6. IF the Enumerator receives HTTP 429 (rate limit) from the target, THEN THE Enumerator SHALL apply a configurable delay between requests, defaulting to 5 seconds, with a configurable minimum of 1 second and maximum of 60 seconds.
7. THE Enumerator SHALL limit the request rate to a maximum of 10 requests per second by default, configurable between 1 and 10 requests per second, to avoid service impact.

### Requirement 4: Exposed Source Maps Verification

**User Story:** As an academic auditor, I want to verify whether Vite source maps are publicly exposed, so that I document the source code exposure risk and provide guidance on how to remove them from production.

#### Acceptance Criteria

1. WHEN the Auditor starts the source map verification, THE Scanner SHALL make HTTP requests to at least 10 common source map paths generated by Vite (e.g., `/assets/*.js.map`, `/assets/*.css.map`), first fetching the application HTML to extract actual asset URLs before constructing `.map` paths.
2. WHEN a source map request returns HTTP 200 with Content-Type containing `application/json` and a response body that is valid JSON, THE Analyzer SHALL confirm the finding as an exposed source map of high severity.
3. WHEN an exposed source map is confirmed, THE Analyzer SHALL extract and document the `sources` field from the `.map` file, recording up to 5 entries and truncating evidence at 200 characters, to evidence the extent of source code exposure.
4. WHEN an exposed source map is confirmed, THE Reporter SHALL generate a finding documenting: endpoint, high severity, evidence (truncated `sources` field excerpt), and remediation guidance (Nginx configuration to block `.map` files).
5. IF no source map is found at the standard Vite paths (HTTP 404 or non-JSON response for all tested paths), THEN THE Scanner SHALL record the result as "not vulnerable" with medium confidence level, indicating that the check covered standard Vite output paths.
6. IF a source map request results in a non-200, non-404 response (e.g., 403, 500) or a connection timeout, THEN THE Scanner SHALL log the error with status code and path, and exclude that path from the "not vulnerable" result without marking it as a confirmed finding.

### Requirement 5: Hardcoded Secrets and Addresses Detection in JavaScript Bundle

**User Story:** As an academic auditor, I want to analyze the JavaScript bundle for hardcoded API keys, private keys, and contract addresses, so that I document the credential exposure risk and provide guidance on managing them via environment variables.

#### Acceptance Criteria

1. WHEN the Auditor starts the JavaScript bundle analysis, THE Scanner SHALL download the `.js` bundle files from the target application; IF the download of any bundle file fails (non-200 response or connection error), THE Scanner SHALL log the failure with the URL and HTTP status code, skip that file, and continue with remaining files.
2. WHEN bundle files are downloaded, THE Analyzer SHALL apply regular expressions to detect patterns of: Ethereum private keys (64 hexadecimal characters prefixed with `0x`), Ethereum contract addresses (42 hexadecimal characters prefixed with `0x`), API keys (patterns such as `apiKey`, `api_key`, `API_KEY` followed by an alphanumeric string of at least 16 characters), and mnemonics (sequences of 12 or 24 words where each word is a member of the BIP-39 English wordlist).
3. WHEN a suspicious pattern is identified, THE Analyzer SHALL record: source file name, approximate line number (derived from character offset), suspected secret type, obfuscated excerpt (first and last 4 characters visible with middle characters replaced by `***`), and critical severity for private keys or high severity for addresses and API keys.
4. WHEN a hardcoded secret finding is generated, THE Reporter SHALL include remediation guidance recommending use of environment variables via Vite's `.env` file with `VITE_` prefix and instruction to not version `.env` production files.
5. IF no suspicious pattern is found in any bundle file, THEN THE Analyzer SHALL record the result as "not vulnerable" with medium confidence level (one of: low, medium, high).

### Requirement 6: CDN CloudFront Bypass Verification

**User Story:** As an academic auditor, I want to verify whether the real server IP can be discovered by bypassing CloudFront, so that I document the server exposure risk and provide guidance on correct firewall rule configuration.

#### Acceptance Criteria

1. WHEN the Auditor starts the CDN bypass verification, THE Scanner SHALL perform passive queries — historical DNS records, certificate transparency logs, and SPF/MX records — to attempt to identify the server origin IP candidates, evaluating at most the first 5 unique IP candidates found.
2. WHEN an origin IP candidate is identified, THE Scanner SHALL make a direct HTTP request to the candidate IP with the `Host` header set to the target domain, with a connection timeout of 10 seconds; IF the connection is refused or times out, THE Scanner SHALL log the result as "unreachable" for that candidate and continue to the next.
3. WHEN the responses via CDN and via direct IP are equivalent — defined as same HTTP status code AND response body size within 10% of each other — THE Analyzer SHALL confirm the finding as a successful CDN bypass of high severity.
4. WHEN a CDN bypass is confirmed, THE Reporter SHALL generate a finding documenting: discovered origin IP, discovery method, evidence of response equivalence (status code and body size comparison), and remediation guidance (configure Security Group/firewall rules to accept HTTP/HTTPS traffic only from CloudFront IP ranges).
5. IF it is not possible to identify any origin IP candidate from passive techniques, THEN THE Analyzer SHALL record the result as "not vulnerable" with low confidence level, indicating that the check was limited to passive techniques.

### Requirement 7: HTTP Security Headers Verification

**User Story:** As an academic auditor, I want to verify the presence and configuration of HTTP security headers, so that I document missing or misconfigured headers and provide the correct configurations for Nginx.

#### Acceptance Criteria

1. WHEN the Auditor starts the security headers verification, THE Scanner SHALL make an HTTP GET request to the target domain with a timeout of 10 seconds and extract all response headers; IF the request fails, THE Scanner SHALL log the error and abort the check with a "check failed" status.
2. THE Analyzer SHALL verify the presence and validity of the following headers: `Content-Security-Policy` (valid if present with at least one directive), `Strict-Transport-Security` (valid if present with `max-age` ≥ 31536000), `X-Frame-Options` (valid if value is `DENY` or `SAMEORIGIN`), `X-Content-Type-Options` (valid if value is `nosniff`), `Referrer-Policy` (valid if present with a non-empty value), and `Permissions-Policy` (valid if present with at least one directive).
3. WHEN a mandatory header is absent, THE Analyzer SHALL record a finding with medium severity and indicate the missing header name.
4. WHEN the `Strict-Transport-Security` header is present but does not contain `max-age` greater than or equal to 31536000 (1 year), THE Analyzer SHALL record a misconfiguration finding with medium severity.
5. WHEN the `Content-Security-Policy` header is present, THE Analyzer SHALL check whether it contains unsafe directives such as `unsafe-inline` or `unsafe-eval` and record a finding with medium severity if found.
6. WHEN the header verification is complete, THE Reporter SHALL generate a finding for each absent or misconfigured header including: recommended configuration as an Nginx directive and security justification.

### Requirement 8: IDOR Verification in Transaction and Profile Endpoints

**User Story:** As an academic auditor, I want to verify the presence of IDOR vulnerabilities in transaction and user profile endpoints, so that I document the unauthorized data access risk and provide guidance on implementing proper access control.

#### Acceptance Criteria

1. WHEN the Auditor provides the Toolkit with a resource endpoint containing a numeric identifier or UUID (e.g., `/api/users/{id}`, `/api/transactions/{id}`), THE Scanner SHALL generate identifier variations (increment by 1, decrement by 1, substitute with a random UUID, substitute with 0, and substitute with a negative integer) and make authenticated requests with the current user's token.
2. WHEN a request with a different identifier returns HTTP 200 with a response body containing a user identifier field (e.g., `id`, `userId`, `user_id`) whose value differs from the authenticated user's identifier, THE Analyzer SHALL confirm the finding as an IDOR of high severity.
3. WHEN a request with a different identifier returns any 4xx HTTP status code, THE Analyzer SHALL record the result as access control functioning correctly for that endpoint and that identifier variation.
4. WHEN an IDOR is confirmed, THE Reporter SHALL generate a finding documenting: vulnerable endpoint, tested identifier, structural excerpt of the response (field names only, no PII values), impact (unauthorized access to other users' data), and remediation guidance (validate that the resource belongs to the authenticated user before returning data).
5. IF the authenticated request returns a non-2xx status code for all identifier variations, THE Analyzer SHALL record the result as "inconclusive" and prompt the Auditor to verify the authentication token is valid.
6. THE Scanner SHALL perform a maximum of 5 identifier variations per endpoint to minimize service impact and avoid generating excessive records in the target system.

### Requirement 9: Business Logic Verification in Deposit and Withdrawal

**User Story:** As an academic auditor, I want to verify business logic vulnerabilities in deposit and withdrawal flows, so that I document parameter manipulation and race condition risks and provide secure implementation guidance.

#### Acceptance Criteria

1. WHEN the Auditor provides the Toolkit with deposit and withdrawal endpoints along with accepted parameters, THE Scanner SHALL test the following parameter manipulations: -1 (negative integer), 0 (zero), 0.000000001 (extreme fractional), the maximum safe integer (9007199254740991), and a string value (e.g., `"abc"`) for each numeric parameter.
2. WHEN an endpoint accepts a negative value in a withdrawal operation and returns HTTP 200 with a response body containing a balance or confirmation field, THE Analyzer SHALL confirm a parameter manipulation finding of critical severity.
3. WHEN the Auditor enables the race condition test, THE Scanner SHALL dispatch exactly 3 simultaneous requests to the withdrawal endpoint with the same authentication token, record all responses received within 10 seconds, and log any requests that result in timeout or connection error.
4. WHEN 2 or more of the 3 simultaneous requests return HTTP 200 with a response body containing a balance or confirmation field, THE Analyzer SHALL confirm a race condition finding of critical severity.
5. WHEN a business logic finding is confirmed, THE Reporter SHALL generate documentation including: endpoint, manipulated parameters and their test values, observed response (status code and relevant field names without sensitive values), severity, and remediation guidance (server-side range validation and use of locks or atomic transactions for financial operations).
6. THE Scanner SHALL record all business logic test requests in the audit log with: ISO 8601 timestamp, HTTP method, endpoint, sent payload (with sensitive values masked), response status code, and response body size.

### Requirement 10: Nuclei Integration for Known Vulnerability Scanning

**User Story:** As an academic auditor, I want to integrate the Toolkit with Nuclei to execute known vulnerability scans, so that I leverage the community template base and identify vulnerabilities efficiently.

#### Acceptance Criteria

1. WHEN the Auditor starts the Nuclei scan, THE Scanner SHALL verify whether the `nuclei` binary is available in the system PATH and, if not, display installation instructions for the detected operating system.
2. WHEN the Nuclei binary is available, THE Scanner SHALL execute Nuclei with the parameters: target domain, template selection by relevant tags (e.g., `cve`, `misconfig`, `exposure`, `headers`), and JSON output format flag.
3. WHEN the Nuclei scan is complete, THE Analyzer SHALL parse the JSON output file and map each Nuclei finding to the Toolkit's standard finding format (summary, severity, confidence, evidence, next steps).
4. WHEN the Analyzer processes Nuclei results, THE Analyzer SHALL deduplicate findings with the same template ID and the same target, keeping only the first occurrence.
5. FOR ALL Nuclei results processed, parsing the JSON output followed by re-serialization SHALL produce an object equivalent to the original (round-trip property).
6. IF the Nuclei execution fails or returns a non-zero exit code, THEN THE Scanner SHALL record the error with the complete stderr and display a message guiding the Auditor to verify the installation and connectivity.

### Requirement 11: Structured Technical Report Generation

**User Story:** As an academic auditor, I want to generate a structured technical report at the end of the audit, so that I document all findings in a standardized and presentable format for the institution.

#### Acceptance Criteria

1. WHEN the Auditor requests report generation, THE Reporter SHALL compile all findings recorded during the audit session into a single document.
2. THE Reporter SHALL structure the report with sections: Cover Page (with audited domain, date, auditor name, and institution), Executive Summary, Findings Table (with ID, title, severity, confidence, and status), and Finding Details.
3. WHEN the Reporter generates the detail of each finding, THE Reporter SHALL include: finding ID, title, severity, confidence level, description, affected endpoint, technical evidence, potential impact, remediation guidance with exact commands or configurations, and references (CWE, OWASP where applicable).
4. THE Reporter SHALL generate the report in both Markdown and HTML formats, where the HTML must be self-contained (inline CSS) to facilitate sharing.
5. WHEN the report is generated, THE Reporter SHALL order findings by descending severity (critical → high → medium → low) and, within the same severity, by descending confidence level.
6. WHEN the report is generated, THE Reporter SHALL include a "Recommended Next Steps" section listing the three highest-priority fixes based on severity and ease of implementation.
7. IF no findings were recorded during the session, THEN THE Reporter SHALL generate a report with an empty findings section and a note indicating that no vulnerabilities were identified in the tests performed.

### Requirement 12: Iterative Flow with Per-Phase Feedback

**User Story:** As an academic auditor, I want the Toolkit to support an iterative flow where I bring results from each phase for analysis, so that I can conduct the audit in a controlled manner and receive contextualized guidance at each step.

#### Acceptance Criteria

1. THE Toolkit SHALL display at the start of each phase: the phase objective, the exact commands to be executed, and instructions on how to collect and provide results.
2. WHEN the Auditor provides results from a phase, THE Analyzer SHALL interpret the results, classify preliminary findings, and present: finding summary, confidence level, estimated severity, next steps, and exact commands for the following phase.
3. THE Toolkit SHALL maintain persistent session state in a JSON file in the working directory, recording: completed phases, accumulated findings, tested targets, and ISO 8601 timestamp of each operation.
4. WHEN the Auditor resumes an existing session, THE Toolkit SHALL load the saved state and present a summary of already completed phases and already recorded findings.
5. THE Toolkit SHALL prioritize execution of low-risk checks (passive reading, header analysis, source maps) before medium-risk checks (active enumeration, IDOR tests) and high-risk checks (business logic tests, race condition).
6. IF the Auditor attempts to execute a medium-risk or high-risk phase before completing the passive discovery phase, THEN THE Toolkit SHALL display a warning indicating that the recommended order is not being followed and request explicit confirmation before proceeding.
